## Задача: свой ключ OpenAI у учителя (мультиарендность оплаты)

**Зачем:** сейчас вся ИИ-обработка (vision-импорт уроков gpt-4o, переводы, картинки gpt-image-1, тренер gpt-4o-mini) идёт через ОДИН платформенный ключ и ограничена белым списком id (`config.uploadAllowedIds = '1,5,29'` в `backend/src/config.js`). Нужно: другой учитель кладёт СВОЙ ключ OpenAI → генерит/обрабатывает за свой счёт, без белого списка.

**Что сделать:**
1. **Хранение ключа.** Поле `users.openai_api_key` (шифровать — напр. `pgcrypto` или app-level AES с секретом из `.env`; в открытом виде в БД не хранить). Миграция в `backend/src/db/migrations/`.
2. **Настройки.** Поле «API-ключ OpenAI» в настройках учителя (маскированный ввод, «проверить ключ» — тестовый запрос к OpenAI). Никогда не отдавать ключ обратно на фронт в открытом виде.
3. **Использование ключа владельца.** В `backend/src/services/claude.js` / `processor.js` / `imageGen.js` — при обработке урока брать ключ **владельца урока** (`lessons.owner_id`), а не платформенный. Если у владельца ключа нет → текущее поведение (белый список / только банк слов без генерации).
4. **Гейты.** `canUpload()` (`backend/src/routes/process.js`) и генерация картинок (сейчас только owner id=1, `IMAGE_GEN_OWNER_ID`) — разрешать, если у учителя есть валидный свой ключ.
5. **Лимиты/учёт (по желанию Павла):** счётчик потраченных токенов на учителя.

**Ключевые файлы:** `backend/src/config.js`, `backend/src/services/claude.js` (создание OpenAI-клиента), `backend/src/services/processor.js`, `backend/src/services/imageGen.js`, `backend/src/routes/process.js` (`canUpload`), `backend/src/routes/courses.js` (upload-pdf), страница настроек учителя.

**Безопасность:** ключ — секрет. Шифрование at-rest, маскирование в UI, не логировать, не возвращать на клиент.
