## Task A: Мульти-загрузка при редактировании урока (как в «Новый урок»)

Файл: `frontend/src/pages/LessonList.jsx`, компонент `EditForm` (строки ~33-130).

**Важное расхождение с описанием задачи** (проверено): `<input type="file">` в
`EditForm` (строка 121) УЖЕ имеет атрибут `multiple`, и `uploadMedia` (строки
70-83) уже принимает массив файлов и отправляет их все разом
(`for (const f of files) form.append('file', f)`). Множественный выбор файлов
из галереи уже работает.

Реальный, узкий недостающий кусок — сравнение с `frontend/src/pages/NewLesson.jsx`
(`DropZone`, строки 264-298): там на КАЖДУЮ фото-зону (учебник/тетрадь) — ДВА
раздельных `<input type="file">`:
1. Обычный `multiple` инпут (выбор из галереи/файлов, несколько сразу) — уже
   есть в EditForm.
2. Отдельный `<input type="file" accept="image/*" capture="environment" multiple>`
   (строка 293 в NewLesson.jsx) за отдельной кнопкой «📷 Сфотографировать» —
   открывает камеру напрямую (сфотографировать сейчас), а не выбор существующих
   файлов. Такого второго инпута/кнопки в `EditForm` СЕЙЧАС НЕТ ВООБЩЕ.

Сделать: добавить в `EditForm` для КАЖДОЙ из двух существующих кнопок источника
(«📘 Из учебника» / «✏️ Из тетради/доски») по паре точек входа — по образцу
`DropZone` в NewLesson.jsx:
- Существующая кнопка → продолжает открывать обычный файловый пикер (`multiple`,
  без `capture`) — уже работает, не трогать логику, только визуально можно
  переименовать в «📁 Файлы» если это яснее (на усмотрение).
- Новая маленькая кнопка/иконка «📷 Камера» рядом с каждой из двух существующих —
  открывает ВТОРОЙ скрытый инпут с `capture="environment"` для прямой съёмки,
  привязанный к тому же `sourceRef` (textbook/extra), загружает через тот же
  `uploadMedia`/эндпоинт.

Итог: 2 источника (учебник/тетрадь) × 2 способа выбора (файлы/камера) = до 4
точек входа, но переиспользуют один `uploadMedia` и один эндпоинт
`POST /lessons/:id/media?source=...` — только фронтенд-виджет меняется.

Если предложенная трактовка не соответствует тому, что имел в виду Павел (два
раздельных способа ДЛЯ ОБОИХ источников, а не 4 отдельные кнопки) — реализуй как
описано выше (это самая безопасная, обратно-совместимая интерпретация: ничего не
убирает, только добавляет недостающую камеру), но явно отметь в отчёте, что это
интерпретация, и как по-другому это можно было прочитать.

## Task B: Кнопка «История» в Читалке — клиентски, `localStorage`, без сервера

Файл: `frontend/src/pages/TextReader.jsx`.

Контекст (уже устоявшиеся места в коде):
- Режим «Двуязычный»: перевод происходит в `handleTranslate` (строки 372-379),
  результат — `setBilingual(res.translations || [])` (строка 378), массив
  `[{original, translation}]`. Языки — `biSrc`/`biTgt` (строки 179-180... сверь
  актуальные номера строк на месте).
- Режим «Разговор»: сообщения копятся в `convMessages` (строка 214,
  `[{id, side, original, translation}]`), добавляются/обновляются в
  `handleConvResult` (строки 400-414). Языки — `convSrcLang`/`convTgtLang`.
  Есть кнопка очистки текущего разговора (`setConvMessages([])`, строка 645).

Сделать (0 обращений к серверу):
1. **Сохранение в `localStorage`** (ключ, например, `'reader_history'`, массив
   объектов `{ts, mode, src, tgt, items}}` где `items` — соответствующий
   `bilingual`-массив или `convMessages`-массив на момент сохранения):
   - Двуязычный: сохранять ОДНУ запись после каждого успешного перевода (сразу
     после `setBilingual(res.translations || [])` в `handleTranslate` — если
     `res.translations` непустой).
   - Разговор: сохранять/обновлять ОДНУ запись на текущий разговор (не на
     каждую реплику отдельно, чтобы не заспамить историю — реши сам, обновлять
     существующую запись сессии по мере пополнения `convMessages`, или
     сохранять снимок при явном действии — очистке/уходе со страницы; опиши
     выбор в отчёте).
   - Обрезать общий список до последних ~50 записей (`array.slice(-50)` или
     аналог), чтобы `localStorage` не разрастался.
   - Вынеси работу с `localStorage` в небольшой набор чистых хелпер-функций
     (get/save/clear) в начале файла или в отдельный маленький модуль — как
     тебе удобнее для читаемости, не усложняя.
2. **Кнопка «🕘 История»** в шапке Читалки (рядом с существующими элементами
   управления/табами режимов, строка ~504-510). По клику — открывает панель со
   списком прошлых записей: дата, языки (флаги/коды), краткий текст (например,
   первая строка/первые ~60 символов). Список читается из `localStorage`
   **только по клику** (лениво) — НЕ на монтировании компонента, НЕ при
   каждом рендере.
3. **Поиск по тексту** в открытой панели — простой `filter` по подстроке (без
   бэкенда).
4. **Кнопка «Очистить историю»** в панели — чистит `localStorage`-ключ и список
   на экране.
5. Ничего не отправляется на бэкенд ни при сохранении, ни при чтении, ни при
   очистке — только `window.localStorage`.

Не трогай `backend/`. Не трогай AI-тренер. Не трогай логику самого перевода/
распознавания речи в двуязычном/разговорном режимах — только добавляй запись в
историю рядом с уже существующими вызовами `setBilingual`/`setConvMessages`.
