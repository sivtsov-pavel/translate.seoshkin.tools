# ТЗ: раздел «Языковой партнёр» (поиск собеседника для совместного изучения)

## 0\. Важное замечание перед стартом

Рекомендую называть раздел не «Знакомства», а «Языковой партнёр» / «Тандем» — и в UI, и в базе. Причина не в формальности: как только в интерфейсе появляется слово «знакомства» \+ фильтр «мальчик/девочка» \+ фото \+ чат между незнакомыми людьми — это по факту дейтинг-механика, а значит и требования другие: возрастной ценз 18+, модерация фото, юридическая ответственность за общение между пользователями. Если в число «Учеников» вашего приложения потенциально входят несовершеннолетние (у вас есть роль «Учитель»/«Класс», это вероятно), такой раздел либо полностью закрывается для них, либо строится по модели «языковой обмен» (Tandem/HelloTalk), где акцент на языке, а не на внешности — это снимает большую часть риска и юридически, и репутационно. Ниже ТЗ спроектировано во втором варианте, но с полным набором фильтров, который вы просили.

## 1\. Пользовательские сценарии

1. В настройках профиля — опциональный тумблер «Искать языкового партнёра» (выключен по умолчанию, off by default — это принципиально для приватности).  
2. При включении — мини-анкета: город, страна, пол (опционально, можно скрыть), дата рождения (для возраста, не отображается напрямую — только диапазон), родной язык, изучаемые языки \+ уровень, короткое «о себе».  
3. Раздел «Партнёры» в сайдбаре: список карточек с фильтрами (город, страна, пол, возрастной диапазон, родной/изучаемый язык, «онлайн сейчас» — на основе `users_last_seen`).  
4. Карточка партнёра: имя, город/страна, языки, статус «онлайн/был(а) N назад», кнопка «Отправить запрос». Прямых контактов (телефон/соцсети) в профиле быть не может — текстовое поле «о себе» фильтруется на предмет ссылок/номеров.  
5. Запрос уходит адресату (push через `web-push`), тот принимает/отклоняет. После взаимного согласия («match») — открывается чат.  
6. Чат — используется существующая таблица/логика `chat`, но с ограничением: сообщение между пользователями возможно только при наличии активного match (или через уже существующий общий чат — уточнить у разработчика, различается ли он логически от преподавательского чата).  
7. Кнопка «Пожаловаться» на профиль/сообщение и «Заблокировать» — доступны в любой момент, не только после match.  
8. Видеосозвон — отдельная фаза (см. §6), не входит в MVP.

## 3\. Матчинг-логика

MVP — не алгоритмический матчинг, а фильтруемый список (проще, прозрачнее для пользователя, ниже риск «AI решил за меня»):

- Базовый фильтр видимости: `partner_search_enabled = true`, не в списке блокировок друг друга  
- Пользовательские фильтры: город, страна, пол, возрастной диапазон, родной язык, изучаемый язык  
- Приоритет в сортировке (не обязательный фильтр): взаимное совпадение языков — мой `native_language` есть в его `learning_languages`, и наоборот — такие карточки выше в списке  
- Сортировка по умолчанию — по `users_last_seen` (сначала недавно активные)

## 4\. Модерация и безопасность (обязательный блок, не опциональный)

- Раздел выключен по умолчанию для всех, включается явным действием в настройках, с коротким объяснением «зачем» и предупреждением не делиться личными контактами  
- Возрастная граница для раздела — минимум 16+/18+ (решить с юристом/владельцем продукта отдельно от общего возраста регистрации в приложении)  
- Автофильтр текста профиля/сообщений на телефоны, email, ссылки на соцсети (regex, простая эвристика на старте)  
- Rate limit на исходящие запросы (например, не больше 20 в сутки) — защита от спама/масс-рассылки  
- Жалоба → попадает в очередь модерации; администратору (роль «Учитель»/Admin) доступен раздел разбора жалоб — список, статус, возможность заблокировать пользователя  
- Блокировка — двусторонняя невидимость (заблокированный не видит профиль и не может написать)  
- Логирование действий модерации (кто разобрал жалобу, когда, решение) — для подотчётности

## 5\. API (`backend/src/routes/partners.js`)

| Метод | Путь | Назначение |
| :---- | :---- | :---- |
| PATCH | `/api/users/me/partner-profile` | включить поиск партнёра \+ заполнить анкету |
| GET | `/api/partners` | список с фильтрами (query: city, country, gender, ageMin, ageMax, nativeLang, learningLang, onlineOnly) |
| GET | `/api/partners/:id` | карточка профиля |
| POST | `/api/partners/:id/request` | отправить заявку |
| GET | `/api/partners/requests` | входящие/исходящие заявки |
| POST | `/api/partners/requests/:id/accept` | принять → создаёт `learning_partner_matches` \+ чат |
| POST | `/api/partners/requests/:id/decline` | отклонить |
| POST | `/api/partners/:id/block` | заблокировать |
| POST | `/api/partners/:id/report` / `/api/messages/:id/report` | пожаловаться |
| GET | `/api/admin/reports` | (роль admin/учитель) очередь жалоб |
| POST | `/api/admin/reports/:id/resolve` | решение по жалобе |

## 6\. Чат и созвон

- Текстовый чат — переиспользовать существующий модуль `chat`, добавить проверку доступа: сообщения между двумя пользователями через partner-чат разрешены только при наличии `learning_partner_matches` между ними.  
- Видеосозвон — в текущем стеке нет медиа-сервера/WebRTC-инфраструктуры, это отдельная технологическая задача (интеграция с сервисом вроде Daily.co/LiveKit или самостоятельный WebRTC). Рекомендация: вынести в отдельную фазу после того, как текстовая часть отработана и есть реальный спрос — не делать частью MVP.

## 7\. Frontend

- `src/pages/Partners/` — `PartnerList.jsx` (фильтры-чипы по образцу тех, что уже есть в «Словаре»/«Уроках»), `PartnerCard.jsx`, `PartnerProfile.jsx`, `RequestsInbox.jsx`  
- Настройки профиля партнёра — новый блок в существующей странице «Настройки»  
- Push-уведомления (`web-push` уже есть) — новая заявка, заявка принята, новое сообщение от партнёра

## 8\. Этапы

**Спринт 1** — анкета \+ список с фильтрами (read-only просмотр, без заявок) **Спринт 2** — заявки, match, чат (на существующем модуле `chat`) **Спринт 3** — модерация: жалобы, блокировки, admin-очередь — обязателен до публичного релиза, не после **Спринт 4 (опционально, по спросу)** — видеосозвон через внешний провайдер

## 9\. Не входит в MVP

Видеосозвон, алгоритмический матчинг/рекомендации, верификация фото, геолокация по GPS (только город/страна текстом).

## 10\. Метрики успеха

Доля пользователей, включивших поиск партнёра; конверсия заявка → match; доля match, дошедших до первого сообщения; количество жалоб на 100 match (следить, чтобы не росло).
